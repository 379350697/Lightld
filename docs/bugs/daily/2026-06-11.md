## Cluster CL-001-solana-execution-signed-intent-idempotency

### GitNexus Keys

| Key | Value |
| --- | --- |
| Components | execution-solana, live-broadcast, signer-boundary |
| Fingerprint | `solana.execution.signed_intent_unverified.idempotency_missing` |
| Status | fixed-local-full-green / deploy pending |

### Summary

The true Solana execution sidecar accepts `/broadcast` payloads after bearer auth and output cap checks, but does not verify the signed order intent or persist idempotency results. This leaves the mainnet broadcast boundary weaker than the local execution sidecar.

### Root Cause

`local-live-execution-server` verifies Ed25519 signed intents and reuses submissions by `idempotencyKey`. `solana-execution-server` parses the same signed intent shape but proceeds to transaction construction without signature verification or a submission store.

### Attempts

| Time | Attempt | Evidence | Result |
| --- | --- | --- | --- |
| 2026-06-11 | Read execution sidecar parity | `src/execution/local-live-execution-server.ts`, `src/execution/solana/solana-execution-server.ts` | Root cause confirmed. |
| 2026-06-11 | RED tests | Invalid signature, unauthorized signer, duplicate idempotency, idempotency conflict tests failed against old Solana sidecar | Expected RED confirmed. |
| 2026-06-11 | Fix | Shared signed-intent verifier plus Solana execution submission store | Targeted tests green. |
| 2026-06-11 | Closure fix | Added pre-broadcast pending reservation, same-key in-process serialization, stale-pending fail-closed handling, and RPC-send unknown-state retention | Concurrent duplicate requests and write-timeout retries no longer double-broadcast. |

### Fix

Extracted shared signed-intent verification, required Solana `/broadcast` verification, and added file-backed idempotency storage with conflict rejection. The store now reserves a pending idempotency record before Solana RPC broadcast, serializes same-key requests in-process, upgrades the record only after result persistence, and fails closed rather than rebroadcasting if a pending reservation remains or an RPC send attempt returns an unknown state.

| `npm run build` | `tsc --noEmit` | pass |

### Acceptance

- Invalid signature, tampered intent, and unauthorized signer are rejected before any Solana RPC broadcast.
- Duplicate `idempotencyKey` with the same signed intent returns the original result without resending transactions.
- Duplicate `idempotencyKey` with a different signed intent is rejected.
- Concurrent duplicate `idempotencyKey` requests are serialized before broadcast.
- A still-pending reservation fails closed without rebroadcasting.
- A failed RPC send attempt keeps the idempotency key pending instead of allowing an unsafe retry broadcast.

### Final Review Guardrails

- Local code/test closure is complete for the single-process sidecar model.
- The file-backed idempotency lock is process-local. Production must run one Solana execution sidecar instance per state directory unless a cross-process store/lock is added.
- Unknown Solana RPC send state intentionally leaves the key pending. Operators must reconcile chain state before manually clearing or replaying a pending reservation.
- Do not mark this cluster deployed until the real systemd sidecar has been installed and verified against the intended environment.

### Summary

Full-suite closure exposed one real recovery bug and three stale test assertions. Pending submissions that had timed out but were not resolved by chain/account evidence were being cleared as failed, while older tests still expected pre-existing behavior for submitted fills, 15s broadcast timeout defaults, and 10s open retry grace.
